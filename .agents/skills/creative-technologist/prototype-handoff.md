# Skill: Prototype Handoff

## Purpose

Use this skill when turning an idea, workflow, demo, or internal tool into something another team can review, use, maintain, or improve.

## Core Principle

A prototype is only valuable if someone else can understand what it does, why it matters, how to test it, and what would be needed to make it production-ready.

## Handoff Requirements

Every prototype should include:

### 1. Purpose

- What problem does this solve?
- Who is it for?
- What workflow does it support?
- What is intentionally out of scope?

### 2. User Flow

- What does the user input?
- What does the system do?
- What does the user review?
- What is the final output?
- Where does the handoff happen?

### 3. System Flow

- Inputs
- Context
- Processing logic
- AI/model layer if applicable
- QA checkpoints
- Outputs
- Storage or export points

### 4. Review Notes

- What works now?
- What is mocked?
- What needs real data?
- What needs stakeholder review?
- What needs legal/security review?

### 5. Production Readiness

- Required integrations
- Required permissions
- Data handling needs
- Security considerations
- Error states
- Audit/logging needs
- Maintenance owner

## Handoff Standard

The receiving team should be able to answer:

- What is this?
- Why was it built?
- How do I run it?
- What problem does it solve?
- What is fake or demo-only?
- What would it take to ship?

If they cannot answer those questions, the handoff is incomplete.
