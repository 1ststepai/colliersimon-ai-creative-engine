# Skill: Repo Presentation

## Purpose

Use this skill to make the GitHub repo itself feel like a polished portfolio artifact.

A hiring team may review not just the live site, but the repository structure, README, commits, docs, and naming.

## Repo Standards

### README

The README should clearly explain:

- What this is
- Why it was built
- What role it is aligned to
- How the prototype works
- How to run it locally
- How to deploy it
- What is intentionally mocked
- What would be needed for production
- That it is a spec/candidate concept, not official client work

### Naming

Use clear names:

- AI Creative Engine
- Campaign Variant Console
- Brand Context Pack
- Human QA Layer
- Production Handoff

Avoid vague names like:

- AI tool
- Demo thing
- Test project
- New app

### Professional Boundaries

Do not imply:

- Collier.Simon hired me
- This was commissioned
- This is official client work
- This uses private Collier.Simon data
- This uses any proprietary client assets

Use:

- Spec concept
- Candidate prototype
- Role-inspired prototype
- Publicly informed concept
- Demo-safe

### Repo Cleanliness

Ensure:

- No secrets
- No .env with real values
- No API keys
- No private files
- No messy scratch files
- No broken links
- No unfinished TODOs unless clearly marked as future work

### Review Impression

A strong repo should communicate:

- I can think in systems.
- I can build.
- I can document.
- I understand handoff.
- I understand AI risk.
- I can communicate with technical and non-technical teams.
- I do not confuse shiny AI output with production-ready work.
